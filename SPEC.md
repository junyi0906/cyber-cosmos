# SPEC.md: Cyber-Cosmos 第二轮进化规格书

## 1. 项目概述

**项目名称**：Cyber-Cosmos (赛博宇宙) - v2.0 "Echoes of Economy"（经济回响）

**项目目标**：
在第一轮基础架构（事件系统、叙事引擎、外交联盟）之上，深化宇宙模拟的经济深度与历史厚度。引入动态经济系统与赛季机制，使文明的发展具备周期性与历史遗产，同时通过增强观测台提升用户对宇宙演化的感知能力。

**核心技术栈**：
- **后端**：Node.js + TypeScript + NestJS (模块化架构)
- **前端**：React 18 + TypeScript + Vite + TailwindCSS + ECharts (数据可视化)
- **数据库**：PostgreSQL (关系型数据存储) + Redis (实时行情/缓存)
- **通信**：REST API + WebSocket (实时推送)
- **测试**：Jest + Supertest

---

## 2. 需求分析与方向选择

基于现有架构，经分析决定实施以下 4 个核心方向，旨在构建闭环的经济与历史循环：

1.  **【星际贸易系统】（核心）**
    *   *理由*：经济是文明互动的强动力。引入资源交易将激活外交系统的实际意义（盟友贸易优惠），并为叙事引擎提供“贸易战/商路阻断”等素材。
2.  **【宇宙赛季机制】（核心）**
    *   *理由*：解决无限模拟的枯燥感。赛季制可以重置文明格局，允许玩家/观察者体验不同的宇宙规则（如“资源匮乏纪元”或“黄金时代”），增加重玩价值。
3.  **【文明遗迹系统】（重要）**
    *   *理由*：连接叙事引擎与地图实体。重大事件（如“超新星爆发”、“帝国崩塌”）不再仅是文本日志，而是转化为地图上的永久印记，供后续文明探索争夺。
4.  **【观测台增强】（体验）**
    *   *理由*：随着数据维度的增加（经济、历史），急需可视化工具。时间线与热力图能让用户直观感知“宇宙的呼吸”。

*(注：暂未选择“宇宙天气预报”，将其降级为事件系统的一种特殊类型，留待后续迭代)*

## 3. 功能列表

### 3.1 星际贸易系统
- **资源定义**：矿石、能源、技术点三种基础资源，各文明有不同的产出与消耗速率。
- **贸易路线**：文明间可建立贸易路线，路线有“稳定性”属性，受距离、外交关系、随机事件影响。
- **动态市场**：全宇宙统一的资源交易所，价格根据供需关系实时浮动（Redis 计算）。
- **贸易协定**：在外交模块新增“通商条约”，需双方同意方可开启直接贸易。

### 3.2 宇宙赛季机制
- **赛季周期**：每个赛季持续固定时间（如模拟时间 1000 年），分为“开拓期”、“扩张期”、“纷争期”、“终焉期”。
- **规则变量**：每赛季生成独特的宇宙常数（如：本赛季能源产出 +20%，但外交信任度恢复 -50%）。
- **赛季结算**：赛季结束时统计文明评分（领土、经济、科技），生成“赛季传奇榜单”，并保留历史数据。

### 3.3 文明遗迹系统
- **遗迹生成**：当发生“文明毁灭”、“泰坦战争”、“科技奇点”等 S 级事件时，在坐标点生成遗迹实体。
- **遗迹交互**：其他文明可派遣舰队进行“探索”、“挖掘”或“争夺”。
- **遗迹效果**：遗迹拥有持久光环（如：该星系科研速度 +10%）或一次性奖励（获得失落科技）。

### 3.4 观测台增强
- **宇宙演化时间线**：可拖动的时间轴，回放宇宙历史关键节点（文明兴衰、遗迹诞生）。
- **文明发展对比图**：多维度雷达图对比各文明当前的综合国力。
- **势力范围热力图**：在星图上以颜色深浅展示各文明的影响力辐射范围。

## 5. 技术方案

### 5.1 后端架构
- **贸易引擎**：采用“代理模型”，每个文明作为一个 Agent，根据资源缺口自动挂单或响应市场订单。价格计算使用简单的线性回归模型：`Price = BasePrice * (Demand / Supply)`。
- **赛季状态机**：使用 Redis 存储当前赛季状态，利用 Redis Pub/Sub 机制广播赛季变更事件，触发全服刷新。
- **遗迹生成器**：监听 `EventSystem` 的 `onMajorEvent` 钩子，当事件权重超过阈值时，调用 `RelicGenerator` 在事件坐标生成实体。

### 5.2 前端架构
- **可视化**：引入 `ECharts` 处理雷达图和折线图；使用 `Canvas` 或 `Leaflet` (自定义坐标系) 绘制势力范围热力图。
- **状态同步**：WebSocket 监听 `MARKET_TICK`（市场行情）、`SEASON_TICK`（赛季推进）、`RELIC_DISCOVERED`（遗迹发现）事件，实现毫秒级 UI 更新。

### 5.3 数据库设计
- **Civilizations 表**：新增 `resources` (JSONB) 字段存储矿石/能源/科技点库存。
- **TradeRoutes 表**：存储贸易路线信息。
- **Seasons 表**：存储历史赛季记录及元数据。
- **Relics 表**：存储遗迹坐标、类型、归属及剩余耐久度。

### 5.4 接口设计
- `GET /api/v2/market/prices`：获取当前全宇宙资源价格。
- `POST /api/v2/trade/routes`：建立贸易路线。
- `GET /api/v2/season/info`：获取当前赛季进度与规则修正。
- `GET /api/v2/relics`：获取全宇宙遗迹列表。
- `WS /ws/observatory`：推送观测台实时数据流。

## 6. 测试方案

### 6.1 单元测试
- **贸易逻辑**：验证供需算法是否正确，价格是否随交易量波动。
- **赛季流转**：模拟时间推进，验证赛季是否按时切换，规则修正是否生效。

### 6.2 集成测试
- **遗迹闭环**：模拟触发“超新星”事件 -> 验证数据库是否生成遗迹 -> 模拟文明探索 -> 验证资源奖励发放。
- **贸易外交联动**：验证盟友间贸易税率是否低于敌对文明。

### 6.3 压力测试
- 模拟 1000 个文明同时进行资源交易，验证 Redis 缓存击穿风险及价格计算延迟。

## 7. 运行与部署

### 7.1 开发环境
```bash
# 启动数据库
docker-compose up -d db redis

# 启动后端
cd backend && npm install && npm run start:dev

# 启动前端
cd frontend && npm install && npm run dev
```

### 7.2 生产部署
- 使用 Dockerfile 分别构建前后端镜像。
- 后端服务建议开启 Cluster 模式以应对 WebSocket 高并发。
- 数据库需配置定时备份任务，防止赛季结算数据丢失。

### 7.3 初始化数据
- 运行 `npm run seed` 生成初始宇宙配置、第一赛季规则及初始文明分布。
